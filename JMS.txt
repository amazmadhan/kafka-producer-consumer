	3. Event-Driven (Kafka, JMS Topic)

Is Kafka based on JMS?
No

JMS = Java API
Kafka = independent platform

| JMS               | Kafka       |
| ----------------- | ----------- |
| API               | Platform    |
| Queue-based       | Log-based   |
| Remove after read | Retain data |


JMS vs Apache Kafka:
| Feature           | JMS (IBM MQ, ActiveMQ)         | Kafka                        |
| ----------------- | ------------------------------ | ---------------------------- |
| Model             | Queue-based                    | Log-based streaming          |
| Message retention | Until consumed                 | Stored for configurable time |
| Replay            | No                          	 | Yes (big advantage)        	|

| Use case          | Banking, transactions          | Analytics, event streaming   |
| Guarantee         | Strong (exactly-once possible) | At-least-once by default     |

JMS = Deliver safely and remove
Kafka = Store everything (Retain data) and allow replay

JMS cannot do this "replay"
Kafka can do this "replay"

Step 2: Consumer comes back

Consumer doesn’t need to manually request, Producer needs to send the message to Consumer somehow.

In short,
Queue is used for point-to-point communication where a message is consumed by only one consumer, typically for task processing like loan approval. 
Topic is used for publish-subscribe where one event is consumed by multiple systems, like sending notifications or triggering analytics after a loan is approved. 
In Kafka, only topics exist, but consumer groups allow it to behave like queues.

How JMS works in Spring Boot?
Spring Boot makes JMS very easy using @JmsListener.

Flows:
	1, Producer sends message → Queue
	2, Broker stores message (like IBM MQ)
	3, Consumer (@JmsListener) reads message

Producer:
@Autowired
private JmsTemplate jmsTemplate;

public void sendMessage() {
    jmsTemplate.convertAndSend("loan.queue", "Loan Created: 123");
}

Consumer:
@JmsListener(destination = "loan.queue")
public void receiveMessage(String message) {
    System.out.println("Received: " + message);
}

What Spring does internally?
	Creates connection to broker (IBM MQ / ActiveMQ)
	Converts object → message
	Handles retries, transactions

Is @JmsListener common for all brokers?
Yes (for JMS-compatible brokers)

Works with:
	IBM MQ
	Apache ActiveMQ

Not directly for:
	RabbitMQ (uses different API)
	Kafka (uses @KafkaListener)

Interview Ready Answer for JMS:
JMS is a Java-based messaging API used for asynchronous communication between systems. It helps in decoupling services by allowing producers to send messages to a queue and consumers to process them independently.

What is Decoupling?
Systems (producers and consumer) don’t depend on each other directly.

Example:
Without JMS (Tightly coupled)
Loan → calls Risk → must be UP

With JMS (Decoupled)
Loan → Queue → Risk reads later

Note: Loan doesn’t care if Risk is down

In Spring Boot, JMS is implemented using JmsTemplate for sending and @JmsListener for consuming messages, typically integrated with brokers like IBM MQ or ActiveMQ.

Compared to REST, JMS is asynchronous and more fault-tolerant since messages are stored in queues even if downstream systems are unavailable.

In contrast, Kafka is a distributed event streaming platform that retains messages and allows replay, making it suitable for high-throughput use cases like analytics.

In banking systems, JMS is preferred for transactional use cases like payments and loan processing due to its reliability, while Kafka is used for event streaming and analytics.
